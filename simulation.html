What is Priority Inversion?

A high-priority task gets blocked waiting for a resource held by a low-priority task. Meanwhile, a medium-priority task preempts the low-priority task — causing the high-priority task to wait indefinitely, even though it should run first.

T3 (LOW) acquires shared resource and starts running
T1 (HIGH) arrives, tries to acquire resource — gets BLOCKED
T2 (MED) preempts T3 and runs freely — T1 is starved!
T1 (HIGH priority) effectively runs last — completely wrong order

Fix 1: Priority Inheritance Protocol (PIP)

When a high-priority task blocks on a resource, the lock-holder temporarily inherits the higher priority. This prevents medium-priority tasks from preempting it. Once the resource is released, the priority is restored.

Simple to implement Doesn't prevent deadlocks Used in POSIX mutexes

Fix 2: Priority Ceiling Protocol (PCP)

Each resource has a pre-defined ceiling priority — the highest priority of any task that could ever use it. When any task acquires the resource, it is immediately raised to that ceiling, preventing all potential preemptors.

Prevents deadlocks too Stronger guarantee May over-boost tasks

Real World: NASA Mars Pathfinder (1997)

The spacecraft kept resetting due to priority inversion between a low-priority meteorological data task holding a shared bus, and a high-priority communication task. The fix was enabling Priority Inheritance on the VxWorks mutex — uploaded remotely from Earth!

Project Overview

The Priority Resource Handling System (PRHS) is an OS project that simulates and solves the classic priority inversion problem in multitasking operating systems.

Scheduling Metrics

The simulator tracks these key metrics for each task:

Waiting Time = Time spent in ready queue waiting to run
Turnaround Time = Finish Time − Start Time
Effective Priority = Current priority (may be boosted by PIP/PCP)
Burst Time = Total CPU time the task needs
